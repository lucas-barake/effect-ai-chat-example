- Entity IDs should be branded with `Schema.brand` and defined in the respective RPC module, not globally. Each domain boundary owns its IDs.
- Tests and code should use `EntityId.makeUnsafe()` to construct branded IDs. Never cast with `as EntityId`.
- All live layers (including `Rpc.toLayer`, service layers, middleware layers) must be explicitly typed as `Layer.Layer<ProvidedServices>`. This catches unsatisfied requirements early. The requirements channel should default to `never` when typed explicitly. Intermediate layers and layers exported for testing should let TypeScript infer the full type.
- Never use `Effect.orDie`. Use `Effect.catchTag` or `Effect.catchTags` to explicitly handle each typed error and convert to a defect with `Effect.die` when appropriate. `Effect.orDie` swallows all errors indiscriminately, hiding errors that should be exposed or handled differently.
- Tests must use real production composition. Do not rebuild or rewire the module dependency graph for tests. Mock only at true external boundaries (third-party APIs, databases, etc.) by swapping the boundary layer. Internal service wiring must be identical to production so tests verify the actual composition, not a test-only replica.
- Frontend tests must render the real component hierarchy. Do not construct a custom component tree for tests.
- Never mock atoms, atom modules, or atom definitions. Always run atoms with their real implementations. Mock only the services/layers atoms depend on by providing test layers through the atom registry.
- No barrel `index.ts` files. Do not re-export from other modules. Each module is the sole source of its exports. Consumers import directly from the module that defines the symbol.
- Atoms that interact with external concerns (RPC calls, HTTP requests, storage, etc.) must depend on injectable Effect services or runtime provided layers, not hard coded globals or direct module side effects. In application code, prefer appropriately scoped `ServiceMap.Service` definitions for distinct concerns. Library provided service boundaries are also acceptable when they are wired through layers and remain swappable in tests via the atom registry, for example `Atom.kvs` backed by a `KeyValueStore` runtime layer. Service methods must be explicit function wrappers (`someMethod: (...args) => client.someMethod(...args)`), not direct property references (`someMethod: client.someMethod`). This keeps atoms testable by swapping layers without mocking the atoms themselves.
